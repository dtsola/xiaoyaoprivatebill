## 1. 总体技术架构

### 架构模式
- **架构类型**：前后端分离 + 本地优先
- **部署方式**：本地部署（用户电脑本地运行）
- **核心特性**：
  - 隐私优先：数据不上传任何服务器
  - 轻量级：无需数据库，使用文件系统存储
  - 快速启动：本地服务秒级启动
  - 跨平台：支持 Windows/macOS/Linux
- **模块化设计**：
  - 前端：页面组件、图表组件、状态管理、API 客户端
  - 后端：API 路由层、业务逻辑层、数据处理层、文件解析器

## 2. 前端技术选型

### 2.1 核心框架

- **Vue 3.4+**
  - **选择原因**：
    - 学习曲线平缓，API 简洁直观
    - 单文件组件（SFC）开发体验好
    - 响应式系统性能优异
    - 中文文档完善，社区活跃
    - 适合中小型项目快速开发
  - **核心优势**：
    - Composition API 让代码更易组织和复用
    - 性能提升，比 Vue 2 快 1.5~2 倍
    - TypeScript 支持良好
    - 体积更小，Tree-shaking 友好
  - **替代方案对比**：
    - **React**：生态更成熟，但学习曲线陡峭，需要额外配置状态管理
    - **Svelte**：编译时框架，性能更好，但生态相对较小

- **Vite 5.0+**
  - **选择原因**：
    - 开发服务器秒级启动
    - HMR 热更新速度快
    - 原生 ESM 支持，无需打包
    - 官方推荐，与 Vue 3 深度集成
  - **核心优势**：
    - 开发体验极佳
    - 构建速度快（使用 Rollup）
    - 开箱即用的 TypeScript 支持
    - 丰富的插件生态

### 2.2 UI组件库

**本项目不使用 UI 组件库**

- **选择原因**：
  - 项目界面相对简单，自定义组件更灵活
  - 避免引入不必要的依赖
  - 保持代码轻量
- **实现方式**：
  - 使用原生 CSS + CSS 变量
  - 自定义可复用组件（StatCard、DatePicker、FilterBar 等）
  - 图表使用 ECharts

### 2.3 状态管理

- **Pinia 2.1+**
  - **选择原因**：
    - Vue 3 官方推荐的状态管理方案
    - API 简洁，学习成本低
    - TypeScript 支持完善
    - 无需 Mutation，简化状态更新
  - **核心优势**：
    - 轻量级（约 1KB）
    - 完美的 TypeScript 支持
    - 支持热模块替换
    - 无嵌套模块，结构更扁平
  - **使用场景**：
    - data.js：存储交易数据、分析结果
    - session.js：管理用户会话状态
    - ui.js：管理 UI 状态（加载、提示等）

### 2.4 网络通信

- **Fetch API**
  - **选择原因**：
    - 浏览器原生 API，无需额外依赖
    - 基于 Promise，语法简洁
    - 现代浏览器全面支持
  - **核心优势**：
    - 零依赖，减少打包体积
    - 性能优秀
    - 支持流式处理

### 2.5 其他前端库

- **ECharts 5.4.3**
  - **选择原因**：
    - 功能强大，图表类型丰富
    - 复用老项目代码，降低迁移成本
    - 性能优秀，支持大数据量渲染
  - **核心优势**：
    - 支持折线图、柱状图、饼图、热力图、桑基图等
    - 高度可配置
    - 响应式设计
    - 中文文档完善
  - **使用场景**：
    - 年度/月度收支趋势（折线图）
    - 分类占比（饼图）
    - 时间分布（热力图）
    - 资金流向（桑基图）

- **Vue Router 4.2+**
  - **选择原因**：Vue 官方路由解决方案
  - **使用场景**：页面路由管理

## 3. 后端技术选型

### 3.1 Web框架

- **Flask 2.0+**
  - **选择原因**：
    - 轻量级，核心精简
    - 灵活性高，可自由选择组件
    - 学习曲线平缓
    - 适合小型项目和快速开发
  - **核心优势**：
    - 微框架，无强制依赖
    - URL 路由装饰器语法优雅
    - 内置开发服务器，调试方便
    - 丰富的扩展生态
  - **替代方案对比**：
    - **FastAPI**：性能更好，自动生成 API 文档，但异步编程复杂度高
    - **Django**：功能全面，但过于重量级，学习成本高
    - **Express (Node.js)**：JavaScript 全栈，但数据处理能力不如 Python

### 3.2 数据库

**本项目不使用数据库**

- **选择原因**：
  - 隐私优先，数据存储在用户本地
  - 账单数据是临时分析数据，无需持久化
  - 使用 Pandas DataFrame 内存处理更高效
  - 简化部署，无需数据库配置
- **存储方案**：
  - 用户上传文件：`backend/data/uploads/`
  - 解析数据缓存：`backend/data/cache/`
  - 临时文件：`backend/data/temp/`

### 3.3 数据处理

- **Pandas (Latest)**
  - **选择原因**：
    - Python 数据处理的事实标准
    - 强大的数据分析和处理能力
    - 支持多种文件格式（CSV、Excel）
    - 适合账单数据解析和聚合分析
  - **核心优势**：
    - DataFrame 数据结构高效
    - 丰富的数据操作 API
    - 内置数据聚合、分组、筛选功能
    - 与 NumPy 深度集成

- **NumPy (Latest)**
  - **选择原因**：Pandas 的基础依赖
  - **核心优势**：高效的数值计算

- **OpenPyXL (Latest)**
  - **选择原因**：支持 .xlsx 格式读取
  - **使用场景**：微信账单 Excel 文件解析

### 3.4 缓存方案

- **Pandas Pickle 缓存**
  - **选择原因**：
    - 简单高效，无需额外依赖
    - Pandas 原生支持
    - 适合单机场景
  - **使用场景**：
    - 解析后的交易数据缓存（永久，直到数据清除）
    - 分析结果缓存（TTL: 会话期间）
  - **升级路径**：如需多用户支持，可考虑 Redis

### 3.5 文件解析

- **自定义解析器**
  - **支付宝 CSV 解析器** (`parsers/alipay.py`)
  - **微信 CSV/XLSX 解析器** (`parsers/wechat.py`)
  - **模块化设计**：便于扩展支持其他账单格式

## 6. 部署技术选型

### 6.1 本地部署

- **用户电脑本地运行**
  - **选择原因**：
    - 隐私优先，数据不上传
    - 用户完全控制自己的数据
    - 无需服务器成本
  - **部署方案**：
    - 组件1：Flask 后端服务（Python 虚拟环境）
    - 组件2：Vue 前端静态文件（构建后由 Flask 服务）
    - 组件3：本地数据目录
  - **核心优势**：
    - 零服务器成本
    - 数据隐私保护
    - 离线可用
    - 无网络延迟

### 6.2 前端构建

- **Vite 构建**（前端生产构建）
  - **安装方式**：`npm run build`
  - **配置优化**：
    - 配置1：代码分割（按路由分割）
    - 配置2：Tree-shaking 去除未使用代码
    - 配置3：资源压缩（Gzip）
  - **输出目录**：`frontend/dist/`

### 6.3 后端运行

- **Flask 开发服务器**
  - **选择原因**：
    - 内置开发服务器，调试方便
    - 适合本地应用场景
  - **运行方式**：
    - 开发：`python app.py`
    - 生产（可选）：Gunicorn / uWSGI

## 8. 技术选型对比总结

### 8.1 前端框架对比

| 框架 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **Vue 3** ✅ | 学习曲线平缓、API 简洁、中文文档完善 | 生态比 React 小 | 中小型项目、快速开发 |
| React | 生态成熟、组件库丰富 | 学习曲线陡峭、需要额外配置状态管理 | 大型项目、团队协作 |
| Svelte | 性能更好、编译时框架 | 生态相对较小 | 性能敏感场景 |

### 8.2 后端框架对比

| 框架 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **Flask** ✅ | 轻量级、灵活、学习曲线平缓 | 功能相对简单 | 小型项目、快速开发 |
| FastAPI | 性能好、自动生成 API 文档 | 异步编程复杂度高 | 高性能 API 服务 |
| Django | 功能全面、ORM 强大 | 过于重量级 | 大型 Web 应用 |

### 8.3 数据处理对比

| 技术 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **Pandas** ✅ | 数据处理能力强、API 丰富 | 内存占用较高 | 数据分析、处理 |

### 8.4 部署方式对比

| 方式 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| **本地部署** ✅ | 数据隐私、零成本、离线可用 | 需要用户自行安装 | 隐私敏感应用 |
| 云部署 | 无需用户安装、便于更新 | 有成本、数据上传云端 | 在线服务 |

### 8.5 存储方案对比

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|

## 10. 技术选型总结

### 核心原则
1. **隐私优先**：数据不上传任何服务器，完全本地处理
2. **简单实用**：选择成熟、简单、易维护的技术
3. **快速开发**：选择学习曲线平缓、开发效率高的技术
4. **轻量级**：避免引入不必要的依赖，保持代码轻量
5. **成本可控**：零服务器成本，用户本地运行

### 技术亮点
✅ **前后端分离**：开发体验好，便于维护和扩展
✅ **本地优先**：数据隐私保护，零服务器成本
✅ **轻量级架构**：无数据库，简化部署
✅ **成熟技术栈**：Vue 3 + Flask + Pandas，社区支持好
✅ **快速启动**：本地服务秒级启动
✅ **跨平台**：支持 Windows/macOS/Linux

### 升级路径

| 阶段 | 用户规模 | 升级方案 |
|------|----------|---------|
| **初期** | 单用户 | 当前方案（本地部署） |
| **成长期** | 多用户 | 添加用户认证 + 云数据库（可选） |
| **扩展期** | 大规模 | 微服务架构 + 分布式存储（可选） |

### 核心技术运维建议

**日常维护：**
- 定期清理临时文件
- 监控磁盘空间使用
- 保持依赖包版本更新
- 备份重要数据

**升级时机：**
- 当需要多用户支持时：考虑添加后端数据库
- 当需要在线访问时：考虑部署到云服务
- 当性能不足时：考虑添加缓存层或优化算法
